4.1.1. General Style Guidelines

Section Summary
PurposeDefines the visual foundation: palette, typography, spacing rhythm, radius, shadows, and component-level surface behavior.
Applies toAll current Nexa public pages and any future marketing or informational pages that share the same design system.
Interpretation ruleWhen an older comment conflicts with the effective current CSS, the effective current CSS wins.

Nexa uses a restrained enterprise-operational visual model for refrigerated food logistics. The base canvas is a blue-tinted near-white, the control hue is a single 221-degree blue family, the dark range is navy rather than black, and most panels are layered through soft tonal gradients, 1px borders, and low-noise shadows. Copy is editorial in structure but left-led in layout. The site does not use decorative color variety, oversized iconography, or symmetrical marketing-card grids as its primary language.

The system is organized around operational clarity. Blue is reserved for action, active state, and proof emphasis. Surfaces stay light until a panel needs to read as a strong proof object, at which point the system switches to dark navy fills with white text. Headings are short, high-contrast, and tightly tracked. Content width is deliberately capped so that large text still reads with urgency rather than banner sprawl.

Design Principles

  • Operational contrast. Primary text on light surfaces uses #151B28. Secondary explanatory copy uses #363F54. Muted copy uses #454F63. Copy lighter than #454F63 is reserved for labels, metadata, and dark-footers only.
  • Single-hue control model. The canonical brand hue is 221deg. New UI colors must derive from the existing blue ladder or from semantic states. A second saturated brand hue is not permitted.
  • Layered surfaces. Standard large panels combine white or pale-blue fills, a 1px border in the 28%-60% alpha range, and a shadow with a long negative spread. Hard flat rectangles without tonal separation are outside the system.
  • Left-led reading flow. Hero copy, section intros, proof blocks, FAQ content, and card stacks default to left alignment. Centered headings are a controlled exception, not the default composition.
  • Action hierarchy. Each section gets one filled primary action at most. A second action may appear only as secondary or tertiary. Two equal-weight filled buttons in the same section are not permitted.

Color Usage Rules

  • Do. Use #2054C5 for filled primary actions, active navigation, badge dots, and one emphasized metric or keyword per section.
  • Do. Keep large section backgrounds below the 300 step of the blue ladder unless the surface is a designated dark proof block.
  • Do not. Mix saturated blue and saturated green as equal-weight adjacent fills inside one viewport.
  • Do not. Introduce pure black, purple, or warm grays. The system is blue-neutral, not generic neutral.

Heading Alignment Rules

  • Left-aligned is default. All page heroes, all 4-column sections, all split layouts, all FAQ content, all route cards, all platform proof blocks, and all solution-detail sections must keep headings left-aligned.
  • Centered headings are allowed only in compact one-column intros. The container must cap at 860px, the intro must not sit beside media, and the section may not contain more than one CTA row directly under the heading.
  • Centered headings are forbidden in page heroes, FAQ, platform module grids, solution-detail pages, and any section that continues into cards or proof columns.

Top-Heavy Card Rule

The current correction pass already centers sparse cards and lowers border weight to avoid top-heavy visual mass. The canonical rule is now fixed: in a standard 24-28px card, the content stack above the first proof element must not exceed 88px. That stack includes eyebrow, title, and introductory copy. If the card needs more than 2 text lines before proof starts, either split the card into two internal columns or turn the proof object into the first row. Do not let the first metric, list item, or form field begin below the vertical midpoint of the card.

Surface Rules

  • Default large cards. White or pale-blue gradient fill, 1px border at 28%-30% alpha, soft panel shadow, and a panel radius of 24.
  • Subtle shadow over heavy borders. When a card sits on a tinted section background, rely on the soft panel shadow and keep the border at or below 30% alpha. Do not use thick borders.
  • Borders are acceptable on buttons, inputs, pills, dropdowns, FAQ links, and light cards. Borders remain 1px only.
  • Borderless wrappers. Grouping shells such as .fit-proof-panel are allowed to be borderless only when their child cards carry the actual surfaces.
  • Tonal fill cards. Hero sidecards, route cards, FAQ groups, module cards, and support blocks should use a pale gradient fill rather than a flat white rectangle whenever they sit on a plain white page area.
  • Darker proof surfaces. Use dark navy surfaces only for the home temperature capability block, dark CTA/contact surfaces, and footer-scale closes. These surfaces must switch all copy to white and internal divider contrast to translucent white.

4.1.2. Web Style Guidelines

Section Summary
PurposeDefines layout mechanics, grid logic, hero patterns, button behavior, forms, motion, and navigation behavior for the live site.
Primary structureThree shell widths, a 4-column base grid, left-led split heroes, and proof-driven closing CTAs.
Operational ruleUse layout to clarify route choice and proof hierarchy, not to create decorative symmetry.

The web system is built on three core layout modes: a standard 1400px content container, a wide 1560px container for hero and proof compositions, and a strict 4-column grid that collapses to 2 columns at 1024px and 1 column at 768px. The page model is consistent: hero, proof sequence, close. Cards do not float as unrelated modules. They are always organized by a clear narrative or operational sequence.

Column Span Rules

  • 2 / 2 split. Use for hero copy plus proof card, editorial content plus proof, or any section where both sides have comparable density.
  • 1 / 3 split. Use only when the left intro is short, capped at about 28-30ch, and the right side contains at least two proof objects. This pattern is already used in home chaos, capabilities, and the compatibility strip.
  • 2 / 1 / 1 distribution. Use for timeline and support grids where a larger introductory or featured card must establish hierarchy before two secondary cards.
  • 3 / 1 distribution. Use only for closing CTA blocks where the headline needs dominance and the action stack occupies a narrow final column.
  • Full-width featured module. The featured platform module spans 1 / -1 and internally re-splits into 1 / 3.

Text Width Rules

  • Home hero content block: 680px max width.
  • Wide inner-page hero content: 1120px max width.
  • Medium inner-page hero content: 980px max width.
  • Home hero headline: 10.5ch max width.
  • Inner-page H1: 11ch max width. Solution-detail H1 may extend to 13.5ch only when the subject label is long.
  • Page-hero lead: 34ch-48ch on split/editorial heroes. FAQ lead extends to 52ch. Closing CTA summary extends to 56ch.

Section Width and Breathing Rules

  • Desktop side breathing room may not fall below 24px. Standard sections should land between 24px and 40px. Wide hero sections may extend to 56px.
  • Full-width sections are allowed only for hero media backgrounds, quiet tonal bands, footer backgrounds, navbar overlays, and modal overlays.
  • Text blocks and card grids are not allowed to run edge-to-edge. They must sit inside one of the three defined containers.
  • Do not create narrow centered bands below 640px width when the section continues into cards. If a section contains proof cards, align the intro left and attach it to the grid.

CTA Hierarchy Rules

  • Maximum CTA count per section: 2.
  • Maximum filled primary CTAs per section: 1.

Microinteraction Guidance

  • Hover. Every interactive element must produce a visible response within 180ms or less. For buttons, combine color change with either shadow or a -1px lift.
  • Click / press. Return the element to neutral position within 120ms. Do not add bounce or elastic easing.
  • Copy action. Button label changes to Copied for 1.5s and then returns to the original label.
  • Invalid action. Show a visible warning or error state immediately. Silent failure is not allowed.
  • Form submit. Use a loading state first, then a visible success or error state. The current form already reveals a success confirmation block after submit.
